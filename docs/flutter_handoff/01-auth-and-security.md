# 01 - Authentication And Security Changes

## ما الذي تغيّر؟

تمت إعادة بناء جزء مهم من المصادقة والأمان في الباكند.

## التغييرات الأساسية

### 1. كلمات المرور
- سابقًا كانت كلمات المرور مخزنة بشكل خام.
- الآن كلمات المرور يتم تشفيرها قبل التخزين.
- عند أول تسجيل دخول ناجح لمستخدم قديم، يتم تحويل كلمة المرور القديمة إلى نسخة مشفرة تلقائيًا.

### 2. Session في الويب
- سابقًا كان النظام يخزن `username` و`password` داخل الـ session.
- الآن الـ session يخزن `user_id` فقط.

### 3. API Login
- سابقًا كان `POST /api/login` فقط ينجح أو يفشل بدون token.
- الآن يرجع `token` يجب على تطبيق Flutter تخزينه محليًا واستخدامه في الطلبات التالية.

### 4. حماية مسارات API
- المسارات المحمية أصبحت تعتمد على التوكن.
- لم يعد مسموح الاعتماد على اسم المشرف المرسل من التطبيق لتحديد هوية المستخدم.

## ما المطلوب من Flutter؟

### بعد تسجيل الدخول
- اقرأ `token` من response.
- خزنه في مكان آمن مثل `flutter_secure_storage` أو أي storage آمن مناسب.
- أرسله مع كل request محمي.

## شكل الهيدر المطلوب

```http
Authorization: Bearer <token>
```

## مثال Login جديد

### Request

```json
{
  "username": "teacher",
  "password": "secret123"
}
```

### Response ناجح

```json
{
  "success": true,
  "message": "تم تسجيل الدخول بنجاح",
  "data": {
    "token": "TOKEN_VALUE",
    "user": {
      "id": 2,
      "username": "teacher",
      "permission": 2
    }
  },
  "errors": {}
}
```

### Response فاشل

```json
{
  "success": false,
  "message": "اسم المستخدم أو كلمة المرور غير صحيحة",
  "data": {},
  "errors": {
    "credentials": [
      "بيانات الدخول غير صحيحة"
    ]
  }
}
```

## نقاط مهمة

- `permission` ما زال موجودًا في response ويمكن لـ Flutter استخدامه لتحديد صلاحيات العرض.
- التوكن هو المرجع الأساسي لهوية المستخدم داخل الـ API.
- لا تعتمد بعد الآن على `admin_name` لتحديد المشرف الحالي.
